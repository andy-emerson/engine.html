## Ranking legend

Every row in both tables carries two independent ratings.

**Urgency** ‚Äî how much other planned work is waiting on this landing:
- **High** ‚Äî one or more concrete next steps can't start, or can't be done safely, until this lands.
- **Medium** ‚Äî later work benefits or is loosely coupled, but nothing is strictly blocked.
- **Low** ‚Äî nothing currently planned depends on this; a standalone improvement.

**Risk** ‚Äî how much already-shipped, working behavior this touches if it goes wrong or drifts:
- **High** ‚Äî touches a subsystem several other shipped features already depend on (the doc model, History, Storage tiers), or can silently corrupt user data.
- **Medium** ‚Äî touches a load-bearing subsystem, but additively (new call sites) rather than modifying existing behavior.
- **Low** ‚Äî isolated or additive; a bug here doesn't ripple into other shipped features.

**Sort:** Urgency first, Risk second (within equal urgency, the riskier item is sequenced sooner ‚Äî deferring it just lets more shipped code accumulate on top of the unstable part). Hard technical prerequisites (X literally cannot be built before Y exists) override the score when the two disagree; noted inline where that happens.

Each row also splits **problem** (why it matters ‚Äî durable) from **solution ideas** (how we might do it ‚Äî a sketch, not a commitment; a good place to park ideas we're not ready to build).

## Agent plan

### The model
The agent is a **collaborator, not a chatbot**. It runs a selectable **Mode** (a protocol), has **Sight** (reads the project), optional **Hands** (acts on cells / conf / assets via tools), on a **Backend** (local WebLLM and/or a remote API), and communicates through **Scaffolding** the app enforces. One system message ‚Äî rebuilt each turn from live state ‚Äî is where Sight lives; tools are the Hands channel; a Mode is data (system prompt + sight/hands level + backend preference), eventually user-authored.

Three driving ideas (from the oracle, kept):
1. **Local + remote collaboration.** Local (WebLLM) is free/fast/private but weak; remote is frontier-quality. Use local as a cheap scout and remote as the closer. Variant: local attempts a step behind a checkpoint; on failure the app reverts and escalates to remote with the failed attempt + error. **A naive version of this (local relays the raw message + full Sight to remote, no real scouting) was actually built and headless-tested this session, then deliberately removed** ‚Äî see Already built and the Local-try‚Üírevert‚Üíremote-escalate row below. The lesson it left: this idea only earns its complexity once local can do something real (attempt an edit, filter context) gated behind actual tools, not as a default toggle state with a no-op local leg.
2. **Modes.** A named protocol = system prompt + config. Assistant (casual), Tutor (one runnable step at a time ‚Äî and in LoveIDE each step is a real cell that shows on the game canvas), Rigor (the scaffolding discipline). Rigid modes need a capable model (remote); a local 1.5B can't hold them.
3. **Scaffolding that reduces miscommunication.** The agent presents a frontier of options the human picks from (never railroads); every claim carries a grade and may not sit above its evidence; deviations are surfaced. Claims are graded by **witnessed actions** ‚Äî enforced by the substrate, not trusted to the prompt.

### How LoveIDE differs from the oracle (important)
The oracle's plan is built around a **DuckDB runtime substrate** shared with a **reactivity** branch. LoveIDE has neither, which *simplifies* the foundation:

- **`main.lua` is already the single source of truth.** There is no separate DB state to serialize ‚Äî **Sight = serialization of the doc model + a conf summary + the love.js Console** (the Variables/Tables static analysis feeds the human-facing panels, not Sight ‚Äî there are no dependency maps or table schemas to share, because those features don't exist). No `_ql_*`, no DuckDB.
- **No reactivity branch.** `main.lua` is one program, not a reactive cell graph, so the "build the substrate once, two branches unblock" argument doesn't apply.
- **Sight is static + Console.** Our Variables/Tables are *static* (parsed source), not runtime values. The agent sees the code + Console (print/errors) but not live game state (e.g. `player.x` at frame 600) unless we add a runtime **debug bridge** into the running love instance (Backlog).
- **Witnessing is complete.** The witnessed signal is "the `.love` booted and ran without a Lua error," via the iframe‚ÜíConsole bridge. Pre-boot failures flip the app's runtime state in `run()`; a post-boot `err` now flips it too (`hookLoveMessages` ‚Äî warns don't, and the iframe stays up so love's error screen remains visible). Headless-tested against the extracted handler; browser confirmation owed.
- **Checkpoint/revert is nearly free.** A snapshot is `serialize(nb.doc)` (+ conf); the History snapshot/restore machinery already exists (and already dedupes + caps itself ‚Äî see the Backlog).

Full chat history for both backends too. Local overflow is a clean `ContextWindowSizeExceededError` surfaced with WebLLM's own exact counts + the remedy (raise the VRAM allocation, or Clear the chat).
- Console is a real, independently-toggleable RHS panel (not the old fixed strip) ‚Äî the RHS/Console seam from the original plan is also done. Full write-up in `design.md`.
- Settings ‚Üí Agentic Coding: "WebLLM Agents" (local) plus "API Agents" ‚Äî add-on-demand remote-agent rows (Anthropic, OpenAI, xAI ‚Äî the three currently testable) with per-agent Commit/Remove, key fields auto-populated from Secrets on render, and boot-time sync (purge secrets/entries for any provider no longer supported, refresh live model lists for the rest). This is the Secrets UI, scoped per-agent instead of a generic named-key manager. Committing a key fetches that provider's real `/models` list (loading/error states included) ‚Äî no hardcoded or placeholder model IDs.
- **Chat is wired to the active agent, group-chat style.** `agent.messages` carries a `from` field for attribution, so the transcript shows which agent answered even across turns where you switched agents. Exactly one agent ‚Äî local WebLLM or a remote provider ‚Äî is ever active: activating one deactivates the other (`agentToggleVram`/`renderAgentRemote`'s activate handler each clear the other's active state; local's disk cache is left alone, just unloaded from VRAM). A brief 2-agent pipeline design (local relays to remote) was built, tested, and then deliberately removed this session once live testing showed it added real cost (VRAM, code complexity, a confusing toggle) for zero functional difference ‚Äî the local leg was a no-op passthrough, so 2-agent mode sent the exact same thing to remote as remote-only. Remote calls (`remoteChat()`) hit Anthropic's `/v1/messages` and OpenAI/xAI's OpenAI-compatible `/v1/chat/completions` directly from the browser, non-streaming for this first pass; response bodies are surfaced on failure, matching the model-list fetch's error handling. Routing tested headless against the real extracted functions. Still Sight-only, no Hands.
- **Chat replies render properly, on the app's side, not the agent's.** Chat bubbles picked up the same `.out-md` markdown styling cell-output already had (they'd been missing it ‚Äî the real cause of an unwrapped-code-forces-horizontal-scroll bug, and, separately, of doubled/uneven paragraph spacing: `.chat-msg`'s `white-space:pre-wrap` ‚Äî needed for plain-text user bubbles ‚Äî was leaking into rendered markdown and preserving marked.js's literal pretty-printing newlines as forced blank lines; `.out-md` now sets `white-space:normal` to stop that). Every fenced code block gets Lua syntax highlighting (`highlightLua()`, a small standalone regex tokenizer ‚Äî not CodeMirror, since a real editor instance being torn down and rebuilt on every streamed token would be expensive and janky) plus a hover-reveal copy button, regardless of whether the agent tagged the fence as `lua` or not (this is a Lua-only IDE, and the offline markdown fallback drops fence-language tags anyway, so language-agnostic-and-always-Lua is the only default that works either way). Headless-tested against real chat content (including against the actual pinned `marked@9.1.6` output).
- **Chat has Copy/Clear (mirroring Console) and cancel/interrupt.** Copy/Clear icons in the AI assistant pane header, same pattern as Console's ‚Äî Clear is also a partial answer to unbounded history cost (see Open Question 4): an abandoned conversation is one click from not being resent every turn. Cancel (Escape while the input is focused, or a Stop button swapped in for Send while generating) is "pulled back," not "leave it sent": your message and the in-progress reply are dropped as if the turn never happened, and your original text is restored to the input ‚Äî chosen specifically because full-history resend means an abandoned turn left in the transcript keeps costing tokens on every later remote call, not just once. Local WebLLM cancellation uses `MLCEngine.interruptGenerate()` (dependency-verified against the real `@mlc-ai/web-llm` package ‚Äî a cooperative flag the generation loop checks between tokens, doesn't throw); remote cancellation uses a plain `AbortController` per turn. Every in-flight generation function checks its own turn's cancellation token (not the shared `agent.cancel` global, which may already belong to a newer turn by the time a stale call's next check runs) before touching `agent.messages` ‚Äî headless-tested against the real extracted functions, including the rapid cancel-then-immediately-resend race (a stale turn's late-arriving reply must not corrupt or reset a newer turn already in flight).
- **VRAM budget tool (Settings ‚Üí WebLLM Agents).** Root cause of Qwen dying after roughly a dozen messages, even on a tiny project: `buildContext()`'s token budget only ever governed main.lua/conf/console ‚Äî the full, ever-growing chat history got appended with zero budget awareness, so a flat `AGENT_CTX_WINDOW=8192` got exceeded on ordinary-length conversations (confirmed by estimating real Qwen replies from the session's own testing: 50-120+ tokens per reply, accumulating fast). Rather than trim history ‚Äî explicitly declined for now ‚Äî this instead measures free VRAM empirically (there's no WebGPU API to query it; verified against the real `@webgpu/types` package: `GPUSupportedLimits`/`GPUAdapterInfo` expose only fixed capability ceilings, never live usage) by filling GPU buffers in chunks via the documented `pushErrorScope('out-of-memory')`/`popErrorScope()` pattern until one fails, then freeing them all (`measureVram()`). A confirm-gated warning runs first (GPU stress + real device-loss risk on some drivers ‚Äî LoveIDE's own pre-existing `DeviceLostError` message already documents OOM causing this in practice, not hypothetical). The result is a snapshot, not a guarantee ‚Äî VRAM is shared system-wide and can go stale the moment another GPU-heavy tab runs ‚Äî so it's meant to be re-run, not trusted forever. Allocation is a slider spanning the full measurement (persisted, auto-corrected if a re-measurement comes back smaller than a stale saved value) with a themed knob matching the existing Paper√óInk `.nd-range` sliders (no browser-default blue) and a dynamic green/amber/red fill with **two-ended zones** (`vramZone` ‚Äî see the dedicated bullet below) ‚Äî the live readout shows the %-of-measurement above the track (the Paper slider's label pattern) and the estimated token window beside it (MB is still what Set persists; it shows in the toast), and the Set button (separate from dragging, so a nudge doesn't persist on every tick; reuses the existing Commit/Remove blue-on-hover pairing) is disabled and reads "Saved" once the slider matches the persisted value, or disabled while it sits in the red zone (still draggable there to look, just not committable) ‚Äî same grayed-out look either way, but the label says which reason applies. `agentPickContextWindow()` then scales `context_window_size` from that allocation against the selected tier's known `vramMB` (confirmed via the installed WebLLM source that the KV cache genuinely does scale with `context_window_size` at reload ‚Äî not a fixed pre-compiled ceiling) ‚Äî an approximation, not an exact formula (none is exposed), backed by real probe/retry: each attempt gets a **fresh engine/device** (a failed reload can itself take the device down, so retrying on a possibly-dead one would just fail again), backing off on failure ‚Äî and if every scaled attempt fails, **throwing an honest re-measure error rather than falling back to a flat default** (no fallback exists anymore; see the bullet below). Headless-tested: the scaling/backoff/honest-throw logic against a mocked engine, and the measurement loop's clear-first/sum/cleanup control flow against a mocked WebGPU device ‚Äî the real GPU calls themselves are browser-only per the claim ladder.
- **VRAM budget is now the sole gate ‚Äî measured, persisted, two-ended.** No flat-window fallback exists: without a measurement + allocation, Qwen cannot be activated at all (toggle disabled, status says why), and `agentPickContextWindow()` throws honestly instead of silently running at a size the user never chose. The measurement persists to localStorage with a timestamp (origin+browser+machine scope ‚Äî exactly a VRAM snapshot's validity domain; readout shows its age, e.g. "measured 2 days ago") so the stress test isn't re-run every visit. The scaling anchor is `VRAM_FIGURES_CTX=4096` ‚Äî dependency-verified against WebLLM's `prebuiltAppConfig`: every published `vram_required_MB` figure is computed at a 4096-token window, so allocation == `vramMB` requests exactly the window the publisher promises (the old 8192 baseline was 2√ó optimistic at that anchor). Slider zones are two-ended (`vramZone`): red below the tier's published `vramMB` (the model can't even be placed), yellow from there to 2√ó it (runs but cramped), then green, then yellow/red again at the top when the allocation would leave under 512/256 MB of the measurement free ‚Äî **absolute reserves, not %**, because what they protect (game preview, browser variance) costs fixed MB regardless of GPU size. Low zones follow the on-disk tier and move on model switch (the persisted allocation is never auto-corrected ‚Äî only its color changes); collision banners state plainly when a tier has no green (or no non-red) position on this machine. A "?" button on WebLLM Agents shows the per-model Min cache / Min VRAM table (straight from `AGENT_MODELS`) with the 4096-window footnote; a tip by Measure suggests measuring while the game runs so the biggest post-measurement consumer is captured by the measurement itself. All zone/anchor/gate/throw logic headless-tested against the real extracted functions (30 checks); the live GPU paths stay browser-only.

### Sequencing decision ‚Äî Sight before Hands

Decided by the user: **perfect Sight before implementing Hands** ‚Äî do one thing
well instead of two things poorly, then build Hands on really good Sight. Hands
still lands before ship. Scoping so far (from the design conversation):

- **Console is the channel.** Runtime-health info reaches the agent through the
  Console, not as separate Sight fields. The runtime-error gap was therefore a
  status-pill/UI fix, not agent work ‚Äî **now fixed** (post-boot errors flip
  `nb.runtime`; see How LoveIDE differs, above).
- **No arbitrary limits, even for local** (supersedes the old Open Question 4 ‚Äî
  see its entry below). **Fully done:** the local-only 14-line Console tail, the
  `AGENT_SIGHT_BUDGET`/`luaDigest` source gate, and the `chars/4` estimator are
  all gone ‚Äî `buildContext()` takes no parameters and both backends get the full
  source + full Console + full history, always. Local overflow fails clean with
  WebLLM's exact counts plus the raise-your-allocation remedy (`agentFailText`).
  Relevance/curation stays deferred.
- Remaining substantive Sight candidates: the **read-only tool loop**
  (reframed as pull-based Sight ‚Äî the agent filling gaps in its own picture;
  zero mutation; the same plumbing later Hands work reuses, with remote
  streaming riding along since it reworks the same `remoteChat()` path), the
  **runtime debug bridge** (Backlog; live game state ‚Äî still an open design
  call), and the **Sight freshness browser test** (Backlog ‚Äî user-reported
  mid-chat staleness; the pipeline is provably fresh headless, so the cause
  is browser-only).

### Open questions
1. **Runtime Sight:** static source + Console only, or invest in a live **debug bridge** into the running game (read `_G`/watches each frame)? See the matching Backlog row.
2. **Witnessing richness:** the boots-vs-errors gap is now fixed (post-boot errors flip the runtime state) ‚Äî is that signal enough to grade claims long-term, or do we want richer ones (canvas screenshot diff, FPS, agent-written assertions)?
3. **Secrets/remote ‚Äî partly resolved.** Storage is the existing encrypted-at-rest layer (device-bound); the Secrets UI question is settled ‚Äî per-agent key fields in Settings ‚Üí Agentic Coding (see Already built), not a generic named-key manager. Still open: is device-bound persistence sufficient, or do we also want a session-only "don't save" mode?
4. **Token/context management ‚Äî resolved.** No arbitrary limits, no pre-flight estimates: the source gate, digest fallback, and `chars/4` heuristic are deleted (a pre-flight count's only job was predicting overflow to gate/trim, and gating/trimming are ruled out). Accuracy comes after the fact instead, where it's exact and free: WebLLM responses carry `usage.prompt_tokens`, and an overflow's own error message carries the exact prompt-vs-window numbers (both dependency-verified against the shipped package) ‚Äî surfaced in chat with the remedy (raise the VRAM allocation, or Clear the chat). Token *display/metering* remains a future nicety (Backlog's cost-of-use row). Relevance/curation stays deferred as being itself an arbitrary limit.
